# PPT Agent 深度研究报告 — Task 1: 主题研究

## 1. 项目概述

**ppt-agent** 是一个基于 Claude Code 插件系统的多智能体幻灯片生成工作流。它将演示文稿的创建过程拆解为 7 个阶段、4 个专业智能体，通过 OpenSpec 任务管理系统协调，输出 SVG 1280×720 格式的 Bento Grid 布局幻灯片，并辅以 Gemini 模型进行质量审查。

**核心定位**：不是一个"一键生成 PPT"的黑盒工具，而是一个**结构化的人机协作工作流**——在关键决策节点（需求确认、大纲审批、最终交付）设置硬停止点，要求用户参与决策。

## 2. 行业背景与竞品格局

### 2.1 AI 演示文稿生成赛道现状（2025-2026）

AI 演示文稿生成已形成三个梯队：

| 梯队 | 代表产品 | 特点 |
|------|---------|------|
| **SaaS 平台** | Plus AI, SlidesAI, Alai, Gamma | 云端 WYSIWYG，模板驱动，集成 Google Slides/PowerPoint |
| **开源框架** | PPTAgent/DeepPresenter (ICIP-CAS), SlideDeck AI | 参考幻灯片分析，编辑操作生成，研究导向 |
| **CLI 工作流** | ppt-agent (本仓库) | Claude Code 插件，多智能体，SVG 原生输出，开发者导向 |

### 2.2 同类方案对比

**PPTAgent/DeepPresenter (icip-cas/PPTAgent)**：
- 中科院信息处理实验室出品，2025 年 12 月发布 V2
- 分析参考 PPT 提取功能类型和内容模式，通过编辑操作生成新幻灯片
- V2 集成了深度研究、自由视觉设计、自主资产创建、文本到图像、沙箱环境等 20+ 工具
- 定位：学术研究框架

**Plus AI / Alai**：
- 商业 SaaS，强调上下文感知（编辑 slide 8 时记住 slide 1-7 的上下文）
- 模板丰富，适合非技术用户
- 局限：封闭系统，定制能力有限

**ppt-agent（本仓库）**：
- 面向开发者的 CLI 原生工作流
- 4 专业智能体分工（研究、内容、设计、审查）
- SVG 原生输出（非 PPTX），可版本控制
- 双模型验证（Claude 生成 + Gemini 审查）
- 与 Claude Code 生态深度集成

### 2.3 差异化定位

ppt-agent 的独特价值在于：

1. **开发者优先**：CLI 驱动、Git 友好、代码即配置
2. **SVG 原生**：矢量输出，无需 Office 依赖，可程序化后处理
3. **双模型质量保障**：利用 Claude 和 Gemini 的互补优势——Claude 擅长结构化创作，Gemini 擅长视觉审查
4. **显式人机协作**：3 个硬停止点确保用户始终掌控方向

## 3. 架构设计哲学

### 3.1 多智能体分工模型

ppt-agent 采用**星型拓扑**架构——主编排器（Lead Orchestrator）位于中心，4 个专业智能体仅与 Lead 通信，不直接互通：

```
                    用户
                     ↕
              Lead Orchestrator
              (commands/ppt.md)
             ↙    ↓    ↓    ↘
    Research  Content  Slide  Review
      Core     Core    Core    Core
```

**设计理由**：
- **简化通信**：避免 N×N 通信复杂度，Lead 作为唯一协调者
- **上下文隔离**：每个智能体在独立上下文窗口运行，不会被无关信息污染
- **故障隔离**：单个智能体失败不影响其他智能体
- **可观测性**：所有通信经过 Lead，天然形成审计追踪

### 3.2 模型选型策略

| 智能体 | 模型 | 理由 |
|--------|------|------|
| Research Core | Sonnet | 搜索和信息提取任务不需要最强推理，Sonnet 性价比最优 |
| Content Core | Opus | 大纲规划需要深层结构化思维和金字塔原理应用 |
| Slide Core | Opus | SVG 生成需要精确的空间推理和像素级控制 |
| Review Core | Sonnet | 审查主要由 Gemini 驱动，Sonnet 负责协调和回退 |

**关键洞察**：将最强模型（Opus）分配给创造性和空间推理密集的任务，将高性价比模型（Sonnet）分配给信息检索和流程协调任务。

### 3.3 质量保障双环路

```
生成环路 (Claude)           审查环路 (Gemini/Claude)
   slide-core                   review-core
   ┌──────────┐                ┌──────────┐
   │ 读取大纲  │                │ 预检自动化 │
   │ + 草稿   │                │ (XML/尺寸) │
   │ + 风格   │                ├──────────┤
   ├──────────┤    slide_ready │ Gemini    │
   │ 生成 SVG │───────────────→│ 质量审查  │
   │ (Bento)  │                │          │
   ├──────────┤                ├──────────┤
   │ 自动验证  │    fixes_json  │ 评分 +   │
   │ (xmllint │←───────────────│ 修复建议  │
   │  等)     │                └──────────┘
   └──────────┘
```

这种"生成-审查"分离的设计借鉴了软件工程中"开发-QA 分离"的思想，利用不同模型的互补视角提升质量。

### 3.4 管道化优化

Phase 6（设计+审查）采用了**管道化**策略：

- Content Core 每生成一张草稿就发送 `draft_slide_ready(index=N)` 信号
- Lead 不等待所有草稿完成，立即为该幻灯片启动 Slide Core
- 滑动窗口限制并行数量为 `min(3, remaining_slides)`

这是对传统"全部生成→全部审查"瀑布模型的改进，将整体延迟从 `T_draft_all + T_design_all + T_review_all` 降低到接近 `T_draft_all + T_single_design + T_single_review`。

## 4. 技术选型深度分析

### 4.1 为什么选择 SVG 而非 PPTX？

| 维度 | SVG | PPTX |
|------|-----|------|
| LLM 友好度 | XML 文本，可直接生成 | 二进制 zip，需要库支持 |
| 版本控制 | Git diff 友好 | 二进制不可 diff |
| 渲染一致性 | 浏览器原生渲染 | 需要 Office 或兼容层 |
| 精度控制 | 像素级精确 | 受模板引擎约束 |
| 生态依赖 | 零依赖 | 需要 python-pptx 等库 |
| 用户转换成本 | 需额外工具转 PPTX | 原生格式 |

**权衡**：SVG 在 LLM 生成场景下有显著优势，但牺牲了与 PowerPoint/Keynote 的直接兼容性。这是一个面向开发者的务实选择。

### 4.2 Bento Grid 设计系统

Bento Grid 是由 Apple 在产品发布会中推广的模块化布局方式，灵感来源于日本便当盒：

- **信息分组**：将内容自然地分配到视觉容器中
- **认知减负**：观众通过结构即可感知信息关系
- **LLM 可生成**：网格布局比自由布局更容易结构化描述

系统定义了 10 种布局组合（single_focus、2-column symmetric/asymmetric、3-column、hero+grid 等），每种都有明确的宽高比约束和排版规则。

### 4.3 认知设计原理集成

ppt-agent 将学术认知科学研究编码为设计约束：

- **Miller 定律**：每张幻灯片最多 5 个信息单元
- **Mayer 多媒体学习原理**：多模态呈现优于纯文本
- **Gestalt 原理**：接近、相似、连续等分组原则
- **三秒测试**：观众应在 3 秒内抓住核心信息

这不是装饰性的文档——这些原理被编码进 prompt，直接影响智能体的生成行为。

## 5. Claude Code 插件生态定位

### 5.1 插件架构

ppt-agent 作为 Claude Code 的插件，遵循标准插件结构：

```
.claude-plugin/plugin.json    → 插件元数据
agents/*.md                   → 智能体定义（Markdown frontmatter）
commands/*.md                 → 用户可调用命令
skills/                       → 技能模块（含脚本、参考、资产）
```

### 5.2 生态集成

ppt-agent 不是孤立系统，它与其他 Claude Code 插件形成协作：

- **agent-reach**：提供跨平台网络搜索能力（Twitter、Reddit、YouTube 等 12+ 平台）
- **topic-research**：上游研究管道，可输出选题报告供 PPT 生成使用
- **visual-content**：下游可视化能力扩展
- **content-production**：内容生产流水线中的一环

### 6.2 完备性

| 方面 | 状态 | 说明 |

| 智能体定义 | ✅ 完整 | 4 个智能体，含信号协议 |

| 质量门控 | ✅ 完整 | 自动化检查 + LLM 审查 + 自适应修复 |
| 交互式预览 | ✅ 完整 | HTML 预览，3 种查看模式 |

| 品牌色覆盖 | ✅ 支持 | `--brand-colors` 标志 |

### 6.3 风险点

1. **Gemini CLI 可用性**：依赖外部 `gemini` CLI 工具，不可用时回退到 Claude 自审，丧失双模型交叉验证优势
2. **SVG 到 PPTX 转换缺失**：对需要传统格式的用户是障碍
3. **无测试覆盖**：工作流正确性完全依赖运行时验证
4. **xmllint 依赖**：SVG 验证依赖系统级工具，可能在某些环境缺失

## 7. 小结

ppt-agent 代表了一种新兴的"开发者优先"AI 内容生产范式：

- **不是取代设计师**，而是为技术用户提供结构化、可控、可版本化的演示文稿生成能力
- **不是单一模型**，而是多智能体分工协作 + 双模型交叉验证
- **不是黑盒生成**，而是人机协作工作流，用户在关键节点保持控制
- **不是商业 SaaS**，而是开源 CLI 插件，深度集成 Claude Code 生态

这是 AI 辅助内容创作从"一键生成"走向"工程化协作"的一个有价值的探索。
